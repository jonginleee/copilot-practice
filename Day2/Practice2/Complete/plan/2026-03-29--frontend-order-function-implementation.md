# 구현 계획: 프론트 주문 기능 적용
현재 홈 화면 메뉴 그리드에 장바구니/주문 흐름을 연결해 고객이 메뉴 선택부터 주문 완료 확인까지 수행할 수 있도록 구현한다. 제품 범위(오프라인 키오스크)와 기존 데이터 흐름(메뉴 조회 -> 장바구니 -> 주문 생성)을 유지한다.

## 아키텍처 및 설계
- UI 구성
  - 주문 화면([src/frontend/kiosk_app.py](src/frontend/kiosk_app.py))에서 메뉴 카드 + 수량 선택 + 장바구니 담기 버튼 제공
  - 우측 또는 하단 고정 영역에 장바구니 요약(항목, 수량, 소계, 총액) 표시
  - 주문 완료 시 주문번호/총액 확인 영역 표시
- 프론트 상태 관리
  - Streamlit `st.session_state`를 사용해 선택 수량, 로딩 상태, 최근 오류 메시지, 주문 완료 상태를 관리
  - 새로고침 시 유실 가능한 임시 상태와 API 원본 상태를 구분해 처리
- 백엔드 연동 계약
  - 장바구니 API: `POST /cart/items`, `PUT /cart/items/{menu_id}`, `DELETE /cart/items/{menu_id}`, `GET /cart`, `DELETE /cart`
  - 주문 API(구현 시): `POST /orders`를 호출해 장바구니를 주문으로 확정
  - 주문 응답: 순차 증가 숫자형 주문번호와 주문 시각 timestamp를 포함
  - 결제 단계: 실제 결제 연동 없이 모의 완료 처리로 고정
  - 오류 코드는 사용자 메시지로 매핑(예: 400 품절, 404 메뉴 없음, 422 입력 오류)
- 계층/책임 분리
  - 화면 파일에는 렌더링과 이벤트 바인딩만 두고, API 호출/응답 변환은 별도 클라이언트 함수로 분리
  - 반복 포맷(가격 표시, 에러 처리)은 공용 유틸 함수로 추출
- UX 제약 반영
  - 터치 사용을 고려해 버튼/입력 컨트롤 크기 확보(최소 44x44px 이상)
  - 주문 핵심 행동(담기, 수량 변경, 주문하기, 초기화)은 한 화면에서 2탭 이내 접근 가능하도록 유지
