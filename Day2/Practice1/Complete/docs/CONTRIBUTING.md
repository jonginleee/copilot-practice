# CONTRIBUTING.md — 햄버거 가게 키오스크 시스템

---

## 1. 코딩 규칙

- 백엔드는 **FastAPI + SQLAlchemy**, 화면은 **Streamlit**을 기본 기술 스택으로 유지합니다.
- 계층 책임을 분리합니다.
  - `routes`: 요청/응답 처리
  - `services`: 비즈니스 로직
  - `repositories`: DB 접근
- 함수/변수/파일명은 의미가 드러나도록 작성하고, 중복 로직은 서비스 계층으로 통합합니다.
- 요청/응답 스키마는 Pydantic으로 명시하고, 모든 입력값은 검증합니다.
- DB 세션/트랜잭션 처리는 일관된 패턴(성공 시 커밋, 실패 시 롤백)으로 구현합니다.
- 하드코딩된 비밀값(API 키, 토큰, 비밀번호) 커밋을 금지하며, 환경변수를 사용합니다.

---

## 2. 테스트 기준

- 테스트 프레임워크는 `pytest`를 사용합니다.
- **FastAPI의 각 기능(메뉴 조회/관리, 장바구니, 주문 생성, 주문 상태 변경, 매출 조회)에 대해 단위 테스트를 작성**합니다.
- 각 기능은 최소 아래 케이스를 포함해야 합니다.
  - 정상 케이스: 유효한 요청에 대해 기대 상태코드와 응답 데이터 반환
  - 예외 케이스: 잘못된 입력, 품절 메뉴, 빈 장바구니, 존재하지 않는 리소스 등
- 테스트는 독립적으로 실행 가능해야 하며, 테스트 DB(SQLite 별도 파일 또는 인메모리)를 사용합니다.
- 신규 기능 PR은 관련 테스트를 반드시 포함하며, 기존 테스트를 깨뜨리면 병합하지 않습니다.

---

## 3. 개발 흐름

1. 이슈 생성 또는 작업 범위를 명확히 정의합니다.
2. 기능 단위 브랜치를 생성합니다. (`feature/*`, `fix/*` 권장)
3. 코드 작성 시 계층 분리 원칙을 지키고, 로컬에서 테스트를 실행합니다.
4. 변경 내용과 테스트 결과를 포함해 Pull Request를 생성합니다.
5. 코드 리뷰 승인 후 `main`에 병합합니다.

PR 본문에는 다음을 포함합니다.
- 변경 목적 및 범위
- API/DB 스키마 영향 여부
- 실행한 테스트 목록 및 결과

---

## 4. 금지 사항

- 테스트 없이 핵심 로직(주문/결제/매출) 변경 금지
- 라우터에 비즈니스 로직 직접 구현 금지
- 운영 데이터 파일(SQLite 원본 DB) 직접 수정 금지
- 불필요한 대규모 리팩토링을 기능 변경 PR에 혼합하는 행위 금지
- 포맷팅 전용 변경과 기능 변경을 하나의 PR에 혼합하는 행위 금지
